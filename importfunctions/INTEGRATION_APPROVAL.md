# 🧩 Інструкція для Antigravity — оновлення блоку «Імпорт бланків узгодження» (Задача #5)

**Контекст:** у поточному GitHub-проєкті лежить **застаріла** копія `approvalImport.ts` — build `approval-v2-20260610` (10 червня), 84 KB, **без OCR**. Канонічна, найновіша версія — build `approval-v2-20260617` (17 червня), 144 KB, **з OCR (tesseract) + розпізнаванням контурів зі сканів + joints**. Вона у цьому пакеті: `approvalImport.CANONICAL.ts`.

**Що це за операція:** контрольована **заміна** старого файлу новим. Я звірив публічний API та типи — вони **сумісні**:
- Функції-експорти ідентичні: `parseApprovalText`, `parseApprovalFile`, `importApprovalFormPdf`.
- `ApprovalImportPreview` — **повністю ідентичний** в обох версіях.
- `ApprovalImportItem` — canonical є **надмножиною**: додає лише одне опціональне поле `joints?`. Жодне поле не видалене й не змінене.
- Споживачі (`FormsPanel.tsx`, `forms/import/ApprovalOverview.tsx`) читають тільки поля, що є в canonical.

Тобто ризик зламати споживачів — низький. Але є три обов'язкові адаптації (нижче).

---

## КРОК 1 — Додати залежності OCR

Канонічна версія використовує `tesseract.js` (їх немає в поточному package.json). Додай у `dependencies`:

```json
"tesseract.js": "^7.0.0",
"@tesseract.js-data/eng": "^1.0.0"
```

Потім `npm install`.

> ⚠️ OCR тягне мовні дані з CDN: `https://cdn.jsdelivr.net/npm/@tesseract.js-data/eng/...`. На проді (Vercel/Supabase) переконайся, що CSP/мережа не блокують jsdelivr — інакше OCR тихо впаде на сканованих бланках. Векторні PDF працюватимуть і без OCR.

## КРОК 2 — Замінити файл

Заміни поточний `src/utils/approvalImport.ts` вмістом `approvalImport.CANONICAL.ts` з цього пакету.

Канонічний файл **вже підготовлений** під GitHub-конфіг:
- переноси рядків нормалізовані CRLF → LF;
- імпорт типів переведено на `import type { ... }` (вимога `verbatimModuleSyntax: true` — інакше білий екран при HMR, як уже було).

Перевір, що ці імпортовані сутності існують у поточному проєкті (мають існувати — стандартні):
`referenceData` (`domain/defaults`), типи `DetailShape, DetailType, EdgeFeature, EdgeProfileSelection, EdgeProfileType, MaterialType, Point` (`domain/types`), бібліотеки `jszip`, `pdfjs-dist`, `xlsx`.

`npx tsc --noEmit` → очікувано чисто. Якщо TS свариться на поле `joints?` у споживачів — воно опціональне, споживачі його не зобов'язані читати; помилок бути не повинно.

## КРОК 3 — Перевірити споживачів

Відкрий `src/components/ui/FormsPanel.tsx` і `src/components/forms/import/ApprovalOverview.tsx`. Вони імпортують `parseApprovalFile`, `ApprovalImportItem`, `ApprovalImportPreview`. Нічого міняти не треба — API той самий. Просто переконайся, що збірка проходить.

## КРОК 4 — Перевірка вручну (користувач)

1. `npm run dev`, **Ctrl+Shift+R**, консоль F12 відкрита (ловимо ESM/Vite-помилки).
2. Імпортуй **векторний PDF-бланк** узгодження → деталі мають розпізнатись (геометрія, розміри, кромки).
3. Імпортуй **сканований/растровий бланк** → має спрацювати OCR (повільніше; перший раз вантажить langdata з CDN). Перевір статуси `OK / Needs review`.
4. Звір кілька деталей із реальним бланком — розміри, сторони, кромки, отвори.

## КРОК 5 (ОКРЕМА ЗАДАЧА, НЕ ЗАРАЗ) — Декомпозиція

Цей файл — **144 KB / 3566 рядків / 113 функцій** = god-object, який поверне в чистий репо те, від чого ви пішли. **Спочатку — щоб працювало (Кроки 1–4). Декомпозицію робити окремою сесією**, коли реально заходитимете в логіку імпорту (це і є зміст Задачі #5).

Цільова структура (орієнтир):
```
src/parsers/approval/
├── index.ts              // фасад: parseApprovalFile, importApprovalFormPdf
├── pdfVector.ts          // витягування векторної геометрії з PDF
├── ocr.ts                // tesseract pipeline (lazy import)
├── imageContour.ts       // розпізнавання контурів зі сканів
├── specTable.ts          // парсинг таблиці-специфікації (xlsx/text)
├── dimensions.ts         // зіставлення розмірів зі сторонами
├── geometry.ts           // побудова points/holes/joints
└── types.ts              // ApprovalImport* типи
```
Робити так само, як ми робили з `export.ts`: по одному модулю, `tsc` між кроками, runtime-перевірка на тих самих бланках до/після, атомарні коміти.
